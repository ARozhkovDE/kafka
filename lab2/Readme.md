# Практическая работа №2: Работа с Kafka

В практической работе №1 вы развернули локальный Kafka-кластер из трёх серверов. Используйте его для выполнения работы №2.

### Шаг 1. Подготовка инфраструктуры
* Создайте топик с **3 партициями** и **2 репликами** через консоль.

### Шаг 2. Разработка приложения
Создайте приложение, состоящее из **1 продюсера** и **2 консьюмеров**. Продюсер будет отправлять сообщения в Kafka-топик (модель **push**).

> **Подсказка:** Хотя продюсер использует асинхронный метод для отправки сообщений, продюсер и консьюмеры работают с потоками данных. Поэтому для них применяйте те же условия, что и при работе с любыми IO-потоками.

#### Стратегии потребления сообщений:
1. **SingleMessageConsumer**:
   * Считывает по одному сообщению.
   * Обрабатывает его и коммитит оффсет **автоматически**.
2. **BatchMessageConsumer**:
   * Считывает минимум по **10 сообщений** за один `poll`.
   * Обрабатывает сообщения в цикле и один раз коммитит оффсет **вручную** после обработки пачки.

> **Подсказка:** Изучите, как работает метод `poll()`, и поэкспериментируйте с настройками `fetch.min.bytes` и `fetch.max.wait.ms`. 
> * `fetch.min.bytes` — определяет минимальный объём данных (в байтах), который консьюмер должен получить за один запрос.
> * `fetch.max.wait.ms` — определяет максимальное время ожидания данных от брокера.

#### Управление смещениями (Commit):
При ручном управлении необходимо вызывать метод подтверждения:
* **Java:** `consumer.commitSync();`
* **Python:** `consumer.commit(asynchronous=False)`
* **Go:** `consumer.Commit()`

#### Требования к работе:
* **Группы потребителей:** Консьюмеры должны работать параллельно и считывать одни и те же сообщения (используйте уникальный `group_id` для каждого типа).
* **Обработка ошибок:** Если возникает ошибка, консьюмеры должны записывать сообщение о ней в логи и продолжать работать.
* **Масштабирование:** Приложение нужно запустить в двух экземплярах с помощью параметра `replicas` в `docker-compose`.

### Шаг 3. Сериализация и десериализация
* Выберите формат данных по желанию.
* Создайте класс сообщения и сериализуйте его перед отправкой в Kafka.
* Выведите отправляемое сообщение на консоль.
* Реализуйте десериализацию сообщений в консьюмерах с выводом на консоль.
* При возникновении проблем с сериализацией/десериализацией выведите ошибку на консоль.

### Шаг 4. Гарантии доставки сообщений
Обеспечьте гарантии доставки сообщений (**At Least Once**):
* Настройте подтверждение доставки в продюсере.
* Используйте параметр `acks` и настройте параметр `retries` в продюсере.
